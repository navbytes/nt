nt — completed-tasks UX exploration

Four directions for how done tasks show up in nt, drawn from the research (Todoist hide-at-bottom, Things 3 Logbook, Kanban collapsed-count, and Anthropic's long-running-agent guidance where completed work = validated context for Claude). Nothing here is built yet — it's to react to. Two audiences: you (review / work-journal) and Claude (recall what we finished). Colors approximate the Tokyo-Night terminal theme.

○ open ◐ doing ✓ done ⊘ blocked / cancelled @tag +project claude / you — who completed it

A · The cheap fix — a “done” count chip (active list stays clean)

Today done is hidden but invisible — you can't tell any exist. Same problem we just fixed for filters. A subtle ✓ 12 done chip top-right signals they're there and one keypress away. Nothing else changes. . reveals them (frame B), 3 opens the Logbook (frame C).

B · . toggled on — done revealed inline, dimmed at the bottom of its group

Todoist's pattern: completed items sink to the bottom of their group, struck-through and greyed, so you keep the context ("I finished this here") without it crowding what's left. The chip flips to a bright ✓ 12 shown so the toggle state is obvious.

C · The Logbook — a dedicated “what got done” view ⭐

A third tab (3, clickable) — Things 3's Logbook. Done & cancelled items grouped by completion date, newest first. The header carries the streak; each row shows who closed it (claude/you). This is your work journal and the exact thing Claude reads to recall recent progress — one view, both audiences.

▎On disk this is just the existing todo.txt completion marker — x 2026-06-06 … src:claude. Grouping by the x date is nearly free since nt already groups by date. Older entries roll off to done.txt via nt archive (cold storage), keeping the Logbook to "recent".

D · The Claude-facing side — nt log as a recall feed

The same completion data, as an append-only CLI feed Claude reads at session start to "get up to speed on what we recently finished" — Anthropic's progress-file pattern. Plain text for the prompt, --json for structured ingest. Completed items stay read-only history (don't rewrite).

▎Pairs with the planned nt recall: recall answers "what do we know about X", log answers "what did we just finish" — together they're the read-side of nt's AI-memory thesis.

Recommended path: ship A (the count chip — tiny, fixes the invisibility) and C (the Logbook tab — highest value, nearly free given date-grouping, serves both you and Claude). Treat B as optional polish and D (nt log) as the fast-follow that makes the same data useful to Claude. The active list never gets more cluttered; "done" lives where it's reviewed, not where you work.
